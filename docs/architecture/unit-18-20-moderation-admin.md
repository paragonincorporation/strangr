# Units 18–20: moderation, admin, and enforcement

Reports accept exactly one server-owned encounter, call, or message context. The API derives the subject from durable participation records, stores a bounded note, and copies at most the reported message's first 500 characters. It never accepts media, SDP/ICE, screenshots, broad history, billing data, or client-selected subject IDs. Idempotency is scoped to reporter plus client request ID. Reporting remains available to restricted accounts and does not end contact unless `leaveAfterSubmit` is true.

The conservative beta reason taxonomy is `sexual_content`, `harassment`, `hate_or_threats`, `minor_safety`, `spam_or_scam`, `self_harm`, and `other`. Minor-safety and self-harm cases enter urgent triage; threats enter high priority. This is an implementation baseline, not launch approval: Safety still owns final taxonomy and response ranges.

Admin endpoints live only under `/v1/admin`, accept configured admin origins, and require a Supabase token with AAL2. Normal reads require authentication no older than eight hours; high-risk changes require reauthentication within five minutes. Database roles are Support, Moderator, Admin, and Superadmin. Navigation never grants permission. Every case/evidence/appeal read and privileged mutation writes an append-only, content-free audit row in the same transaction; an audit failure rolls back the operation. Evidence reveal is explicit and purpose-bound. The admin client uses session storage isolated by the separately deployed admin origin.

Sanctions support warnings, matching/contact restrictions, suspension/full ban, profile removal, and verification challenge. Permanent sanctions require evidence references and Admin permission. Active sanctions are combined with account state at HTTP and realtime boundaries; enforcement removes queues and live leases, publishes revocation, closes sockets, and is rechecked by heartbeat. Appeals are owner-scoped, idempotent per sanction, and cannot be reviewed by the sanctioning operator. Expiry is reconciled by the worker; reversals and successful appeals append history instead of deleting it.

Migration `0006_moderation_admin_sanctions.sql` creates reports, cases, evidence ownership, roles, immutable audits/actions, sanctions, and appeals. Before production, Security must configure a dedicated admin Supabase project/provider flow and approve the phishing-resistant MFA factor; Product/Safety must approve the reason and sanction matrices.
